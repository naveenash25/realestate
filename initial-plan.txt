 Real Estate Platform — Requirements & Architecture Plan

 Context

 A no-middleman, direct-to-owner real estate lead generation platform for India.
 Listing is free. Property owners only pay per genuine buyer enquiry — no
 subscription, no lock-in. Buyers contact owners directly and can negotiate discounts
  without any agent in between.

 ---
 Business Model

 Strategy: No middleman, no subscription — pure pay-per-lead. Listing a property is
 free. Owners only pay when genuine buyer enquiries come in.

 ┌──────────────────┬──────────────────────────────────────────────────────┐
 │      Layer       │                        Detail                        │
 ├──────────────────┼──────────────────────────────────────────────────────┤
 │ Listing fee      │ Free — anyone verified can list                      │
 ├──────────────────┼──────────────────────────────────────────────────────┤
 │ Lead charge      │ Prepaid wallet, charged per lead in real-time (+GST) │
 ├──────────────────┼──────────────────────────────────────────────────────┤
 │ Minimum recharge │ ₹500 (until first 1,000 properties onboarded)        │
 ├──────────────────┼──────────────────────────────────────────────────────┤
 │ Free lead buffer │ 3 leads per wallet-empty cycle (cyclic reset)        │
 ├──────────────────┼──────────────────────────────────────────────────────┤
 │ No balance       │ Property hidden after 3 buffer leads until recharge  │
 ├──────────────────┼──────────────────────────────────────────────────────┤
 │ Converted lead   │ No extra charge — only the initial enquiry is billed │
 ├──────────────────┼──────────────────────────────────────────────────────┤
 │ Daily lead cap   │ Configurable per listing by owner                    │
 └──────────────────┴──────────────────────────────────────────────────────┘

 Differentiation from competitors (e.g. NoBroker): No monthly lock-in. Owners pay
 only for results. Zero upfront commitment.

 Revenue stream: Per-lead charges + future ad integrations (Meta, Google).

 ---
 User Roles

 1. Buyer / User (Demand Side)

 - Can browse property details before login
 - Login: Google OAuth OR Mobile OTP only (no email/password)
 - After login: can submit an enquiry on any property
 - Cannot see owner contact details before submitting an enquiry
 - Each submitted enquiry is recorded (time, date, property, phone number)

 2. Property Owner (Supply Side)

 - Must be verified before listings go live
 - Registration paths:
   - Employee or representative with valid ID
   - Authorized candidate (direct)
   - Initial phase: platform's Management Trainees register on their behalf
 - One owner can have multiple properties
 - Prepaid wallet system — must recharge to keep listings active beyond 3 leads
 - Receives real-time lead notifications (only if wallet is recharged)
 - Can see leads only if wallet has balance

 3. Platform Staff / Admin

 - Verifies property owners before they go live
 - Management Trainees assist with early property registrations
 - Initial phase: relaxed verification; stricter controls enforced as platform scales

 ---
 Property Listing Rules

 - Images: Up to 10 per property
 - Videos: Instagram Reels links + YouTube links (no direct upload initially)
 - Only verified owner properties are displayed publicly
 - Pre-launch properties: allowed but caution note shown to buyers
 - Non-RERA approved properties: allowed but caution note shown to buyers

 ---
 Wallet & Listing Visibility Logic

 3-lead buffer resets every time the wallet hits zero (cyclic, not lifetime):

 Each property tracks: buffer_leads_used (resets on each wallet-empty cycle)

 State machine per property:
   WALLET EMPTY + buffer_leads_used < 3:
     → Listing visible, leads accepted (no charge, buffer consumed)

   WALLET EMPTY + buffer_leads_used = 3:
     → Listing HIDDEN until owner recharges

   WALLET RECHARGED:
     → buffer_leads_used resets to 0
     → Listing visible again, leads charged from wallet

   WALLET DRAINS TO 0 AGAIN:
     → buffer_leads_used resets to 0 (new 3-lead buffer cycle starts)
     → Listing still visible for next 3 leads (no charge)
     → Hidden again after those 3

 Example cycle:
 1. Owner recharges ₹500 → wallet active, leads charged per enquiry
 2. Wallet hits ₹0 → 3 free buffer leads start (listing stays visible)
 3. 3 buffer leads used → listing goes hidden
 4. Owner recharges again → buffer resets, listing visible, charges resume
 5. Repeat

 On each new enquiry (wallet has balance):
 1. Buyer submits enquiry form (owner contact hidden before submit)
 2. Enquiry saved to DB (timestamp, property_id, buyer_phone, buyer_name)
 3. Lead cost (pricing matrix) deducted from owner wallet in real-time (+GST)
 4. Owner notified via:
   a. Real-time push to owner dashboard (Supabase Realtime)
   b. SMS + WhatsApp message with buyer contact details

 Daily lead cap: Set by the owner per property. Once cap is hit, property shows
 "enquiries paused for today."

 ---
 GCP Deployment Plan

 All services run as Docker containers on Cloud Run — serverless, auto-scales to
 zero, pay per request.

 GCP Project
 ├── Cloud Run
 │   ├── nextjs-frontend        (Next.js, scales 0→N)
 │   ├── nodejs-api             (NestJS, scales 0→N)
 │   └── fastapi-ml             (Phase 2 only)
 ├── Cloud SQL
 │   └── postgres-main          (PostgreSQL 15, private VPC)
 ├── Memorystore
 │   └── redis-main             (Redis 7, private VPC)
 │                               ↑ use Upstash if budget is tight
 ├── Cloud Storage
 │   └── property-media-bucket  (images, videos)
 ├── Cloud CDN                  (serves frontend + images)
 ├── Cloud Armor                (WAF, DDoS protection)
 ├── Secret Manager             (API keys, DB creds)
 └── Artifact Registry          (Docker images for CI/CD)

 Cost control levers:
 - Cloud Run scales to zero when idle — no idle cost
 - Cloud SQL can use shared-core (db-f1-micro) in early phase, upgrade live
 - Upstash Redis (pay-per-request) is cheaper than Memorystore until ~50k req/day
 - Switch to Memorystore when sustained Redis load justifies reserved instance

 CI/CD: GitHub Actions → build Docker image → push to Artifact Registry → deploy to
 Cloud Run (zero-downtime rolling deploy)

 ---
 Phase 1 Foundations That Enable Phase 2 (No Rewrites)

 1. pgvector extension installed on Cloud SQL — embedding vector(1536) column on
 properties, null for now. Phase 2 runs a backfill job only.
 2. Internal event bus (NestJS EventEmitter) — every key action emits an event
 (enquiry, recharge, listing published). Phase 2 FastAPI ML service subscribes via a
 dedicated queue topic — no hooks to add.
 3. Versioned REST API (/api/v1/) — ML service and future clients consume v1.
 Breaking changes go to v2 without touching Phase 1 consumers.
 4. BullMQ queue already running — SMS/WhatsApp use it in Phase 1. Phase 2 adds ML
 job types to the same queue.
 5. NestJS modular structure — ListingsModule, LeadsModule, WalletModule,
 NotificationsModule are isolated. MlProxyModule is added in Phase 2 to call FastAPI
 — zero changes to existing modules.
 6. Node.js → FastAPI HTTP contract defined in Phase 1 — ML service interface
 (request/response schema) is documented as an OpenAPI spec even before FastAPI is
 built. Phase 1 Node.js has a stub MlService that returns defaults — Phase 2 replaces
  the stub with real HTTP calls.
